KAM-6 · Phase 1 — Architecture Design & API Contract
Raw OpenAPI

ewc-backend Architecture

Extracting all business logic from ewccli into a standalone, containerized backend service — with the CLI becoming a thin HTTP client and a first-class API for AI agents.

📐 ADR-001 · Proposed 📅 2026-07-03 🏷️ Issue KAM-6 ⏱️ Effort: S (~1 person-week)
1 Context & Goals

Why we are extracting business logic out of the CLI into a standalone backend.

ewccli is a Python CLI that manages European Weather Cloud (EWC) infrastructure: OpenStack VMs, Kubernetes CRDs (DNS records, S3 buckets), hub-item deployments (Ansible playbooks), and authentication. Today the CLI stores long-term OpenStack credentials in local files and calls the cloud directly; there is no identity provider, no OIDC, and no backend.

Today all business logic — OpenStack SDK calls, Ansible execution, Kubernetes CRD management, credential handling — lives inside the CLI process. This creates four core problems:

🔓

Credential exposure

OpenStack application credentials are stored on the user's local disk (~/.ewccli/profiles) in plaintext. Rotation forces every user to re-run ewc login.

⏳

No shared state

Long-running deployments (VM creation + Ansible, DNS waits up to 20 min) block the terminal. If the CLI is interrupted, the operation is lost with no recovery path.

🔁

Duplicated logic

Any automation, CI pipeline, or future web UI must reimplement the same orchestration logic — or shell out to the CLI.

🤖

Not agent-friendly

AI agents cannot reliably drive a Rich/Click terminal UI; they need structured, predictable HTTP endpoints with machine-readable schemas and error codes.

🎯 Target
A standalone ewc-backend service holds all business logic. ewccli becomes a thin client calling the backend API. AI agents, CI pipelines, and future UIs all consume the same versioned OpenAPI contract.

4 Service Layering

A strict three-layer architecture: API → Service → Backend Clients, with cross-cutting job engine and secret manager.

Responsibility boundaries

LayerResponsibilityKey Dependencies
API Layer
FastAPI		HTTP routing, pydantic validation, JWT validation (JWKS), scope enforcement, request/response envelopes, RFC 9457 error mapping, Idempotency-Key dedup.FastAPI, Starlette, Pydantic v2
Service Layer
Business Logic		Orchestration logic, job submission, credential resolution, progress tracking, result aggregation, retry/cleanup policy.Domain models, Job Engine interface
Backend Clients
Cloud SDKs		OpenStack VM lifecycle, Kubernetes CRD management, Ansible playbook execution. One client per cloud backend.openstacksdk, kubernetes, ansible-runner
Dependency rule
Dependencies point inward only: API → Service → Backend Clients. The service layer never imports FastAPI; backend clients never import service logic. This keeps each layer independently testable.
5 Framework & Technology Choices

FastAPI is the recommended backend framework. Here is the rationale and alternatives.

✅ Decision: FastAPI

  • Shared schema layer. ewccli already depends on pydantic 2.x; FastAPI is built on pydantic, so models can be shared without a translation layer.
  • Automatic OpenAPI. FastAPI generates an OpenAPI 3.x spec from route definitions — the spec in this repo is the contract both the CLI and agents consume.
  • Native async. async/await is essential for long-running operations (VM provisioning, Ansible, DNS polling) that must not block the event loop.
  • Dependency injection. FastAPI Depends maps cleanly to the service-layering model.
  • Ecosystem. Uvicorn/Starlette, SQLAlchemy async, structured logging, OpenTelemetry.

Alternatives considered

AlternativeVerdictReason
FlaskRejectedSynchronous by default; no built-in OpenAPI generation; needs flask-smorest/apispec to reach parity.
Django REST FrameworkRejectedHeavier; ORM coupling; less natural fit for a stateless API service.
aiohttpRejectedCapable but smaller ecosystem; no automatic schema generation.

8 CLI ↔ Backend Communication

REST for synchronous operations; async job submission + polling/SSE for long-running work.

⚡ Synchronous (REST)

Read operations and fast writes return immediately.

  • GET /v1/servers — list VMs
  • GET /v1/hub/items/{name} — show hub item
  • POST /v1/auth/token — exchange code for token

HTTP 200 · < 1s

🔄 Asynchronous (Job)

Long-running operations return a job reference (202); client polls or streams.

  • POST /v1/servers → job_id
  • POST /v1/hub/items/{name}/deploy → job_id
  • GET /v1/jobs/{id} — poll status
  • GET /v1/jobs/{id}/logs — SSE stream

HTTP 202 · minutes

CLI UX preserved
The thin CLI handles the async flow transparently: submit → poll every 5s (or --follow for SSE streaming) → render Rich progress bar → display result. If interrupted, ewc jobs show <id> resumes.
9 Authentication & Authorization

Three trust boundaries: User/CLI → Backend, Agent → Backend, and Backend → OpenStack/K8s.

👤

User / CLI → Backend

OAuth 2.0 Authorization Code + PKCE. Browser-based; a new pattern — the current CLI has no IdP integration.

🤖

Agent → Backend

OAuth 2.0 Client Credentials grant with scoped API tokens.

☁️

Backend → Cloud

OpenStack app credentials & kubeconfigs held in a server-side secret store (proposed: OpenBao), rotated server-side — never on the client.

Current state (ewccli) — long-term credentials in files

Today the CLI holds long-term OpenStack application credentials in files on the user's local disk (~/.ewccli/profiles, plaintext INI). The user obtains application credentials manually from the EWC web portal and provides them directly to ewc login via CLI arguments, environment variables (OS_APPLICATION_CREDENTIAL_ID / OS_APPLICATION_CREDENTIAL_SECRET), interactive prompts, or an existing ~/.config/openstack/clouds.yaml. There is no identity provider, no OIDC, and no token ceremony. The CLI persists the credentials to disk and, on every later command, reads them and calls OpenStack / Kubernetes directly — there is no backend.

Experiment only An OpenBao JWT login path was explored as an experiment but is not part of the production CLI and is not how any user authenticates today. It informed the target secret-store decision below; it is not the current production flow.

Problems: long-term credentials on disk in plaintext; rotation forces every user to re-login; no per-user revocation; no client-side expiry tracking (only discovered on 401/403).

Token lifecycle

  • Access tokens are short-lived (15 min default, configurable).
  • Refresh tokens are long-lived (8 h) and stored in the OS keychain, not plaintext on disk.
  • The CLI auto-refreshes before expiry. --no-browser mode prints the auth URL for headless/SSH.
  • Key difference from today: the token is a backend token, not an OpenStack credential. Cloud credentials never touch the client.

Backend as OAuth2 resource server

The backend validates incoming JWTs using Keycloak's public keys (JWKS) — no database lookup needed. Validation checks: signature, exp, iss, aud, and scope claims. The backend then resolves the caller's tenant, reads the matching OpenStack credential from the secret store, and opens a short-lived server-side connection.

🤖 Agent-friendly design

  • OpenAPI as tool definition. The spec doubles as the agent's tool schema — agents discover endpoints, parameters, and responses programmatically.
  • Predictable envelopes. Every response uses { data, meta: { request_id } }; every error uses RFC 9457 problem details with machine-readable code.
  • Idempotency. Idempotency-Key header lets agents retry safely after network failures without duplicate resources.
  • Async-first. Long operations return 202 + job_id; agents poll or subscribe to SSE.

🔐 Agent authorization

  • Registration: admin registers agent as OAuth2 client in Keycloak (or scoped API token via admin API).
  • Scoping: client assigned specific scopes based on what the agent may do.
  • Token acquisition: client_credentials grant → scoped access token.
  • Enforcement: every request carries Bearer token; backend validates + enforces scopes per endpoint.
  • Revocation: API tokens are revocable; client secrets stored in secret manager.
11 OpenStack Credential Rotation

Target (proposed): credentials live in a server-side secret store — never on client disk — and the backend rotates them automatically. OpenBao is the recommended store but is not in production today (the current CLI keeps long-term credentials in local files and has no OpenBao integration).

Credential storage

OpenStack application credentials are stored in OpenBao KV2 (proposed), not the database or client disk. OpenBao is recommended because the team has experimental familiarity with it; the service layer talks to an abstract SecretStore port so it is swappable.

Secret-store alternatives considered

  • Kubernetes Secrets + Sealed Secrets / SOPS — no new component if already on K8s; GitOps-friendly. No dynamic secrets; hand-built rotation.
  • Cloud managed secret manager (e.g. Barbican) — fully managed, native to OpenStack. Vendor lock-in; control-plane latency.
  • PostgreSQL + envelope encryption (KMS) — fewest new components; transactions with job state. Must build key rotation, audit, and policy from scratch.

Expiry detection
If OpenStack returns 401/403 for a credential, the backend marks it expired in OpenBao, returns CREDENTIAL_EXPIRED (HTTP 401), triggers an immediate rotation attempt, and alerts ops if rotation fails.

Long-running operations run as async jobs — submitted, tracked, streamed, and persisted.

Why async?

14 Logs & Progress Streaming

Workers publish to Redis pub/sub; API bridges to SSE clients. Logs are also persisted for later polling.

sequenceDiagram

  participant Q as Redis Queue
  participant W as Job Worker
  participant DB as PostgreSQL
  participant PS as Redis PubSub
  C->>API: POST /v1/servers (Idempotency-Key)
  API->>Q: enqueue job
  API->>DB: persist job (pending)
  API-->>C: 202 { job_id }

  W->>DB: update job (running)
  W->>PS: publish progress / log events
  C->>API: GET /v1/jobs/{id}/logs (SSE)
  API->>PS: subscribe

  API-->>C: SSE stream (real-time)
  W->>DB: update job (completed + result)
  C->>API: GET /v1/jobs/{id}
  API-->>C: 200 { status: completed, result: {...} }

🏗️ Architecture

  • Queue: Redis-backed job queue (RQ or ARQ). API workers enqueue; job workers dequeue.
  • Persistence: Job state (status, progress, result, error) in PostgreSQL — survives restarts.
  • Streaming: Workers publish log/progress to Redis pub/sub. API bridges pub/sub → SSE. Logs also persisted to PostgreSQL for polling.

🔁 Retry & idempotency

  • Transient failures (503, network timeout): retried up to 3× with exponential backoff.
  • Permanent failures (quota exceeded, invalid image): no retry; marked failed with details.
  • Idempotency-Key: if a job with that key exists & is active → return existing; if terminal → return original result; else create new (202). Safe retries, no duplicates.
  • Worker concurrency: one job per worker; horizontal scaling via HA on rq_queue_length.
CLI integration
Submit → poll GET /v1/jobs/{id} every 5s → render Rich progress bar → (optional) --follow opens SSE → display result on completion. If interrupted: ewc jobs show <id> or ewc jobs list to resume.

17 Consequences

Trade-offs of the target architecture, with mitigations.

✅ Positive

  • Centralized credentials: OpenStack app creds live in a server-side secret store (proposed: OpenBao), not on laptops. Rotation is invisible to CLI users.
  • Resumable operations: long-running jobs survive CLI disconnects; users reconnect and check status.
  • Single source of truth: all business logic in one service, testable in isolation, with a versioned OpenAPI contract.
  • Agent-ready: AI agents discover, plan, and execute via structured HTTP + OpenAPI — no terminal scraping.
  • Multi-consumer: future web UIs, CI pipelines, and orchestration tools all consume the same API.

⚠️ Negative

  • Infrastructure overhead: backend requires PostgreSQL, Redis, OpenBao, container orchestration.
  • Network dependency: CLI requires network access to backend; offline operation no longer possible.
  • Latency: extra HTTP hop per operation (mitigated by co-location with OpenStack/K8s).
  • Migration effort: ewccli must be rewritten as thin client; existing profiles need migration.

Mitigations

  • The backend is containerized and can run on a single node initially; components split out as scale demands.
  • The CLI can cache read responses (server lists, hub items) for short-term offline use.
  • A migration tool converts existing ~/.ewccli/profiles entries to backend-managed profiles.
18 Alternatives Considered

Options that were evaluated and rejected, with reasoning.

1. Keep all logic in the CLI Rejected

Does not solve credential exposure, resumability, or multi-consumer access. The CLI remains the only integration point.

2. CLI + shared library (no service) Rejected

Package business logic as a Python library that both the CLI and other tools import. Does not solve credential management, job persistence, or agent access — library consumers must still manage their own OpenStack/K8s connections and credential lifecycle.

3. Serverless functions (Lambda/Functions) Rejected

Ansible playbooks and OpenStack operations can run for many minutes; serverless timeouts (typically 15 min) are too tight for some operations (DNS check timeout is 20 min). Cold starts add latency to interactive operations; debugging is harder.

4. GraphQL instead of REST Rejected

REST + OpenAPI is the lingua franca for AI agent tool-use. GraphQL adds query-language complexity that does not benefit the resource-oriented, CRUD-like operations of this domain. The OpenAPI spec doubles as the agent tool definition.
